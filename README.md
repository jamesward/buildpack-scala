# buildpack-scala

A [classic Heroku buildpack](https://devcenter.heroku.com/articles/buildpack-api)
for Scala apps built with **sbt 2** and packaged with the
[sbt-native-packager](https://github.com/sbt/sbt-native-packager)
`JavaAppPackaging` plugin.

It runs `./sbt stage` and turns the resulting `universal/stage` directory
into the slug. The exact path under
`target/out/jvm/scala-<version>/<project>/universal/stage` is **derived from
sbt** (via `./sbt 'show Universal / stagingDirectory'`) — there is nothing
version-specific or project-specific hardcoded.

## Requirements

- The app has a project-local `./sbt` wrapper at the repository root
  (sbt 2.x publishes one in every new project).
- The app's `build.sbt` enables `JavaAppPackaging` (which provides the
  `Universal / stage` task).
- A JDK is on `PATH` during the build. Chain
  [heroku/jvm](https://github.com/heroku/heroku-buildpack-jvm-common) before
  this buildpack:

  ```sh
  heroku buildpacks:add heroku/jvm
  heroku buildpacks:add https://github.com/jamesward/buildpack-scala
  ```

  Or, equivalently, declare them in an `app.json` at the repo root (used by
  Heroku Button, Review Apps, and `heroku create --manifest`):

  ```json
  {
    "buildpacks": [
      { "url": "heroku/jvm" },
      { "url": "https://github.com/jamesward/buildpack-scala" }
    ]
  }
  ```

  Order matters in both forms: `heroku/jvm` must come first so the JDK is
  on `PATH` before `./sbt` runs. The top-level `buildpacks` array in
  `app.json` drives `bin/compile` + `bin/release` on every deploy. The CLI
  is for already-existing apps; `app.json`'s top-level `buildpacks` only
  applies on app *creation* (Heroku Button, Review App, manifest-create).

## What the slug looks like

After `bin/compile` runs, the build directory is replaced with the contents
of the staged `universal/stage` directory. A handful of names are preserved
across that rewrite — the user's `Procfile` and any slug-side contributions
made by buildpacks that ran *before* this one (e.g. heroku/jvm bundles a
JDK at `.jdk/` and a PATH/JAVA_HOME script at `.profile.d/jdk.sh`):

```
.            <-- slug root
├── bin/
│   ├── <app-name>           # the start script generated by sbt-native-packager
│   └── <app-name>.bat
├── lib/
│   └── *.jar                # all runtime classpath jars
├── .jdk/                    # JDK, preserved from heroku/jvm
├── .profile.d/              # env scripts (e.g. .profile.d/jdk.sh from heroku/jvm)
├── .heroku/                 # generic Heroku slug-side metadata, preserved
└── Procfile                 # preserved from your repo (if any)
```

Without preserving `.jdk/` the dyno would start with no Java on `PATH`, since
heroku/jvm's `.profile.d/jdk.sh` exports `PATH=$HOME/.jdk/bin:$PATH` and
`$HOME/.jdk` would no longer exist.

The set of preserved names lives in the `PRESERVE` array in `bin/compile`.
Add to it if you chain another buildpack that contributes additional
slug-side directories.

## Procfile

Because the slug root is the staging directory, **all paths in your `Procfile`
must be relative to that root**, not to your repo. For an app whose
`name := "hello-zio-http"`, the Procfile is:

```
web: bin/hello-zio-http
```

If you omit the `Procfile`, the buildpack's `bin/release` step picks the first
executable script in `bin/` (excluding `*.bat`) as the default `web` process.

## How the staging dir is derived

`bin/compile` runs sbt twice, in two separate invocations:

1. `./sbt stage` — sbt-native-packager's task that produces the staged
   `universal/stage` directory.
2. `./sbt 'show Universal / stagingDirectory'` — sbt's built-in `show`
   command prints the absolute path of `Universal / stagingDirectory`
   (a setting from sbt-native-packager) to stdout.

The compile script grabs that second invocation's output and pulls the
path out of it with a strict regex — it looks for a line that matches
`^[info] /<path-with-no-spaces>$`, which is what `show` emits for a
`File`-typed setting and which doesn't match sbt's other path-bearing
log lines (e.g. `[info] loading project definition from /...`).

We deliberately use two cold sbt boots (instead of injecting a custom
task or piggy-backing both commands on a single boot) because a custom
task injected via a `*.sbt` file has been observed to silently not run
in some sbt 2 environments — including Heroku's heroku-26 stack. `stage`
and `show` are core sbt commands that don't depend on us injecting
anything, so they work everywhere.

## Multi-project builds (`SBT_PROJECT`)

If your repo is a multi-project sbt build and a non-root subproject is
the one whose `Universal / stage` should drive the slug, set the
`SBT_PROJECT` config var to its sbt project ID:

```sh
heroku config:set SBT_PROJECT=server
```

`bin/compile` will then issue:

1. `./sbt server/stage`
2. `./sbt 'show server / Universal / stagingDirectory'`

instead of the unscoped versions. When `SBT_PROJECT` is unset the
buildpack uses sbt's current (root) project, which is the right default
for single-project builds.

The value must be a valid sbt project ID (alphanumerics + underscores —
no hyphens or dots). For an explicit project definition like
`lazy val server = project.in(file("server"))`, the ID is `server`. For
implicit root projects sbt derives an ID from the directory name and
sanitizes it, so explicit `lazy val` definitions are recommended when
you intend to use `SBT_PROJECT`.

## Caching

The following Coursier / Ivy / sbt directories are persisted in `CACHE_DIR`
between builds, so subsequent deploys do not re-download dependencies:

- `coursier/v1/` (Coursier cache, used by sbt 2 for resolution)
- `ivy2/`        (Ivy home — kept in case any plugin still uses it)
- `sbt/boot/`    (sbt launcher boot dir)
- `sbt/`         (sbt global base)

The buildpack sets `COURSIER_CACHE` and `SBT_OPTS` accordingly before invoking
`./sbt`.

## Heroku CI

This buildpack supports the [Heroku CI testpack
API](https://devcenter.heroku.com/articles/testpack-api) via
`bin/test-compile` and `bin/test`. To run your sbt tests on Heroku CI, add
an `app.json` to your repo, e.g.:

```json
{
  "environments": {
    "test": {
      "buildpacks": [
        { "url": "heroku/jvm" },
        { "url": "https://github.com/jamesward/buildpack-scala" }
      ]
    }
  }
}
```

### How CI caching works (and why it differs from deploy)

`bin/test-compile` produces a *test slug* that is then mounted into every
parallel test dyno that runs `bin/test`. Heroku does **not** pass `CACHE_DIR`
to `bin/test`, so any sbt cache left in `CACHE_DIR` would force every test
dyno to re-resolve every dependency from Maven Central.

To avoid that, `bin/test-compile`:

1. Creates `BUILD_DIR/.heroku-sbt-cache` and points sbt at it
   (`COURSIER_CACHE`, `sbt.boot.directory`, `sbt.ivy.home`, `sbt.global.base`).
2. Warms it from `CACHE_DIR/sbt-cache` if present, so cross-CI-run caching
   still works.
3. Runs `./sbt Test/compile`.
4. Copies the cache back into `CACHE_DIR/sbt-cache` for the next CI run.
5. Leaves `.heroku-sbt-cache` inside `BUILD_DIR` so it ends up in the test
   slug.

`bin/test` then uses the same cache root and runs `./sbt test`. Its exit code
is the test result.

### Test reports

Heroku CI looks for JUnit-style XML reports in `$HEROKU_TEST_RUN_OUTPUT_DIR`
(usually `/app/test-reports`). The default sbt test output is plain text;
to surface structured results, add a JUnit reporter to your build, for
example:

- For [MUnit](https://scalameta.org/munit/), set
  `Test / testOptions += Tests.Argument(MUnitFramework, "+l")` and use
  `sbt-junit-xml-reports` or write reports directly via munit's
  `--report-junit` arg.
- For [ScalaTest](https://www.scalatest.org/), add
  `Test / testOptions += Tests.Argument(TestFrameworks.ScalaTest, "-u", "/app/test-reports")`.

The buildpack itself doesn't impose any test framework.

## Local smoke test

There's a runnable smoke test in `test/smoke.sh` that copies the
`~/projects/hello/hello-zio-http` project into a temp dir, points the
buildpack at it with a temp `CACHE_DIR`, and verifies the slug layout.

```sh
./test/smoke.sh
```

## Layout

```
buildpack-scala/
├── README.md
├── bin/
│   ├── detect        # succeed if build.sbt + ./sbt are present
│   ├── compile       # ./sbt buildpackStage and rewrite BUILD_DIR as the slug
│   ├── release       # emit default_process_types from bin/<exec>
│   ├── test-compile  # ./sbt Test/compile, leave source tree intact for tests
│   ├── test          # ./sbt test (exit code = test result)
│   └── util/
│       └── sbt-env.sh   # shared cache + SBT_OPTS setup
└── test/
    ├── smoke.sh             # local end-to-end test of compile/release
    ├── smoke-subproject.sh  # verifies SBT_PROJECT scoping (uses a stub sbt)
    └── ci-smoke.sh          # local end-to-end test of test-compile/test
```
